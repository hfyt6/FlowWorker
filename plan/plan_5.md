# 第五步：群聊系统实现

## 目标
实现群聊功能，支持多AI对话者参与同一会话，包含完整的防循环调用机制和消息路由系统。

## 具体任务

### 5.1 群聊会话管理
- [ ] 更新 SessionService 支持群聊：
  - 创建群聊会话（指定多个AI参与者）
  - 添加/移除会话参与者
  - 获取群聊会话详情（包含参与者列表）
  - 群聊会话列表查询
- [ ] 更新 SessionsController：
  - POST /api/v1/sessions/{id}/participants - 添加参与者
  - DELETE /api/v1/sessions/{id}/participants/{pid} - 移除参与者
  - GET /api/v1/sessions/{id}/participants - 获取参与者列表
- [ ] 更新前端会话管理：
  - 创建群聊时选择多个AI参与者
  - 会话详情页显示参与者列表
  - 支持动态添加/移除参与者

### 5.2 消息路由服务（MessageRouterService）
- [ ] 实现消息路由核心逻辑：
  - 解析消息中的@提及（@AI名称 或 @AI_ID）
  - 确定目标AI对话者列表
  - 管理调用令牌（每轮初始3个）
  - 跟踪调用深度（最大深度5）
- [ ] 实现路由策略：
  - 并行路由：多个被@的AI同时响应
  - 顺序路由：按提及顺序依次响应
  - 智能路由：根据角色和上下文选择响应顺序
- [ ] 实现响应链管理：
  - AI响应中解析@提及
  - 检查深度和令牌限制
  - 决定是否继续路由

### 5.3 防循环调用机制
- [ ] 实现调用深度控制：
  - 单次用户消息触发的最大深度：5层
  - 每层深度记录到 Message.CallDepth
  - 达到最大深度时自动终止
- [ ] 实现调用令牌机制：
  - 每轮对话初始令牌数：3个
  - 每个AI响应消耗1个令牌
  - 令牌耗尽时停止自动响应
- [ ] 实现互斥响应机制：
  - 同一轮次中，AI响应不会触发其他AI的自动响应
  - 只有显式@提及才触发响应
- [ ] 实现超时机制：
  - 单轮对话总超时：5分钟
  - 单个AI响应超时：2分钟
  - 超时后自动终止本轮

### 5.4 群聊消息处理

- [ ] 实现群聊消息历史：

  - 显示消息发送者信息（名称、头像、角色）
  - 支持按参与者筛选消息
- [ ] 更新 MessagesController：
  - 支持在群聊中发送消息
  - 支持@提及参数
  - 群聊流式响应SSE端点

### 5.5 前端群聊界面

## 预期成果
- 完整的群聊会话管理功能
- 消息路由服务能够正确处理@提及
- 防循环调用机制有效防止无限循环
- 群聊界面支持多参与者显示和交互
- @提及功能正常工作
- 群聊状态可跟踪和管理

## 验收标准
- 能够创建群聊会话并添加多个AI参与者
- 用户@提及AI时，被@的AI能够正确响应
- 调用深度达到5层时自动停止
- 调用令牌耗尽时停止自动响应
- 单轮对话超过5分钟自动终止
- 前端能够正确显示所有参与者的消息
- @提及功能能够正确解析和触发响应
- 停止生成按钮能够中断正在进行的响应链

## 技术要点

### 消息路由流程
```
用户发送消息（包含@AI1 @AI2）
    │
    ▼
MessageRouterService 解析@提及
    │
    ▼
检查调用令牌（剩余: 3）
    │
    ▼
为每个被@的AI创建响应任务
    │
    ▼
AI1 响应（消耗1令牌，深度: 1）
    │
    ▼
解析AI1响应中的@提及
    │
    ▼
检查深度（< 5）和令牌（> 0）
    │
    ▼
继续路由或终止
```
